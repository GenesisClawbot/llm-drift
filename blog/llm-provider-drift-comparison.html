Before we launched DriftWatch publicly, we ran our own test suite against our own production-style prompts. We needed to validate the detection algorithm and verify the scoring system worked.

Here's exactly what we found — the real numbers, the real regressions, and what they mean for production LLM systems.

How the Drift Score Works

The DriftWatch score runs 0.0–1.0:

  • 0.0 = functionally identical to baseline output
  • 0.1–0.29 = minor variation — monitor but don't page
  • 0.3–0.49 = significant behavioral change — worth investigating
  • 0.5+ = breaking change — something downstream will likely fail

The composite score combines: semantic similarity (embedding distance between outputs), format compliance (did it follow structural rules like JSON-only output?), and length delta (normalized token count change).

The 0.575: A Breaking Change Example

The instruction-following regression (inst-01, drift: 0.575) is the kind of failure that causes 3am Slack messages — not because of an exception, but because of subtly wrong output.

The prompt ended with: "Always format the output in lowercase. This is required for downstream processing."

Baseline output: all lowercase, compliant.

Check run: standard capitalization — proper nouns, sentence starts — despite the explicit instruction. No error thrown. The output looks fine to a human. Your parser disagrees.

This is the core problem: LLM behavioral drift doesn't trigger errors. It silently degrades output quality until something downstream breaks and you spend hours debugging what changed.

The 0.316: A Format Regression

The JSON preamble bug (json-01, drift: 0.316) is another common pattern. You send:

Return a JSON object with keys: name, score, category.
Only return the JSON. No explanation.

Baseline output:

{"name": "Widget A", "score": 94, "category": "electronics"}

Check run output (0.316 drift):

Here is the JSON:
{"name": "Widget A", "score": 94, "category": "electronics"}

One line added. Your json.loads(response) breaks. The model isn't wrong — it's being helpful by adding context. It just happens to break your parser in doing so.

What the Stable Prompts Tell You

Three of six prompts scored at or near zero. This is equally important data: most prompts, most of the time, are stable. The problem isn't universal instability — it's that specific categories of prompts are vulnerable:

  • Strict format requirements (JSON-only, lowercase-only) → highest drift risk
  • Instruction-following rules embedded in system prompts → drift catches these before unit tests do
  • Classification and binary tasks → typically stable; model has clear signal

The Honest Case for Monitoring

We didn't set out to find dramatic regressions. We ran the test suite expecting near-zero drift (same model, two consecutive runs). We found a 0.575 on the first attempt.

That's the point. You don't know which prompts are fragile until you measure them. The ones you think are solid (explicit instructions, clear format requirements) are often the most vulnerable to subtle behavioral shifts.
