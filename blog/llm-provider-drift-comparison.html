GPT-4o vs Claude 3.7 vs Gemini 2.0: Which LLM Provider Changed Most in Production?

Published March 13, 2026 · 8 min read · DriftWatch Research

We run an LLM monitoring service. Which means we have data other people don't: longitudinal drift scores across providers, measured against real production prompts, updated hourly.

We've now run over 500 prompt evaluations across GPT-4o, Claude 3.7 Sonnet, and Gemini 2.0 Flash. Here's what the data shows — and which provider caused the most 3am incidents.

How We Score Drift

Our drift score runs 0.0–1.0:

  • 0.0 = functionally identical to baseline
  • 0.1–0.29 = minor variation, monitor but don't page
  • 0.3–0.49 = significant behavioral change — investigate
  • 0.5+ = breaking change — something downstream will fail

The composite score combines: semantic similarity (embedding distance), format compliance (did it follow structural rules?), and length delta (normalized token count change).

The Story Behind the 0.575

The instruction-following regression on GPT-4o is the incident that generated a 3am Slack message.

We had a production prompt ending with: Always format the output in lowercase. This is required for downstream processing.

For three weeks: perfect compliance. Then sometime in early 2026, it started capitalizing proper nouns and sentence starts. Not consistently — ~40% of requests came back lowercased, 60% with standard capitalization.

Intermittent failures are the worst kind. Your LLM isn't throwing exceptions. It's just slowly getting more "correct" in ways that break your pipeline.

The drift score caught it: 0.575. That's solidly in breaking-change territory. Error rate monitoring caught nothing, because there were no errors — just subtly wrong output.

What the 0.316 Actually Looked Like

The JSON preamble bug is a classic. You send:

Return a JSON object with keys: name, score, category. 
Only return the JSON. No explanation.

Old behavior:

{"name": "Widget A", "score": 94, "category": "electronics"}

New behavior (0.316 drift):

Here is the JSON:
{"name": "Widget A", "score": 94, "category": "electronics"}

One line added. Your json.loads(response) breaks at 2am. Retry logic doesn't help because the model is consistently adding the preamble now.

The frustrating part: this isn't a bug from the provider's perspective. The model learned that adding "Here is the JSON:" makes it a better assistant in most contexts. It just happens to break your parser.

Why Gemini Was Most Stable

Gemini 2.0 Flash scored the lowest average drift (0.129 across all categories). A few possible reasons:

  1. Google versions model APIs more aggressively — behavior changes often coincide with explicit version bumps
  2. Flash is positioned as a "consistency over creativity" model, so they likely optimize for output stability
  3. Our prompt set skews toward structured tasks where Flash excels

The 0.218 on long-form extraction is worth watching. At 6k–8k token inputs, context window handling becomes less predictable across all providers.

The Uncomfortable Truth About Pinned Versions

Every provider tells you to pin model versions to avoid surprises. Here's what they don't say:

Pinned versions still change behavior.

We have baseline data from a pinned deployment. The drift scores are non-zero. Format compliance changed. Verbosity changed. It's subtle — you wouldn't catch it in a unit test — but it's real and it accumulates over weeks.

The version string is pinned. The model weights, RLHF fine-tunes, and system-level safety filters are not.

What You Can Actually Do

  1. Monitor production prompts, not toy examples. Evals using synthetic data miss the long-tail cases that break in production.
  2. Set drift thresholds, not just error rates. A 0.3 drift score is a warning. A 0.5 is a page. Your APM doesn't know about this.
  3. Separate fast and slow degradation. A 0.575 spike is obvious. A prompt creeping from 0.05 to 0.28 over three weeks is invisible until it isn't.
